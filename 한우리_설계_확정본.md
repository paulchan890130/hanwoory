## 2. 사이드바 접기/축소 방식

### 2-1. 두 가지 상태 정의

**확장 상태 (기본, 220px):**
```
┌──────────────────────┐
│  🌻 한우리           │  ← 로고 + 브랜드명
│  ─────────────────── │
│  한우리 사무소       │  ← 현재 로그인 사무소명
│  ─────────────────── │
│  🏠 홈 대시보드      │  ← 아이콘 + 텍스트
│  👤 고객관리         │
│  📋 업무관리         │
│  💰 일일결산         │
│  📊 월간분석         │
│  📄 문서자동작성     │
│  📷 OCR 스캔         │
│  📚 업무참고         │
│  🔍 통합검색         │
│  📝 메모             │
│  💬 게시판           │
│  ─────────────────── │
│  ⚙️ 관리자           │  ← is_admin만 표시
│                      │
│  [◁ 접기]           │  ← 하단 접기 버튼
└──────────────────────┘
```

**축소 상태 (아이콘만, 64px):**
```
┌────────┐
│   🌻   │  ← 로고만
│ ────── │
│   🏠   │  ← 아이콘만 (호버 시 툴팁으로 메뉴명 표시)
│   👤   │
│   📋   │
│   💰   │
│   📊   │
│   📄   │
│   📷   │
│   📚   │
│   🔍   │
│   📝   │
│   💬   │
│ ────── │
│   ⚙️   │
│   [▷]  │  ← 확장 버튼
└────────┘
```

### 2-2. 동작 규칙

- 접기/펼치기 상태는 `localStorage`에 저장 (새로고침 후에도 유지)
- 축소 상태에서 메뉴 호버 → 우측에 툴팁 표시 (메뉴명)
- 고객관리, 문서자동작성, 통합검색처럼 넓은 화면이 필요한 페이지 진입 시 → 자동 축소 제안 없음 (사용자 선택 유지)
- 활성 메뉴: 골드 텍스트 + 좌측 4px 골드 선(accent bar)
- 축소 상태에서 활성 메뉴: 아이콘에 골드 배경 원형

## 3. 통합검색 — 완전 설계

### 3-1. 정보구조 (IA)

```
통합검색 (Unified Search)
│
├── 검색 대상 (Search Sources)
│   ├── A. 고객 데이터          [tenant 격리 필수]
│   ├── B. 진행업무             [tenant 격리]
│   ├── C. 예정업무             [tenant 격리]
│   ├── D. 완료업무             [tenant 격리]
│   ├── E. 일정                 [tenant 격리]
│   ├── F. 단기메모             [tenant 격리]
│   ├── G. 장기/중기 메모       [tenant 격리]
│   ├── H. 업무참고/업무정리    [tenant 격리]
│   ├── I. 게시판               [권한 정책별]
│   ├── J. 매뉴얼 데이터        [공용 또는 tenant]
│   └── K. 문서작성 템플릿 메타 [공용 읽기 전용]
│
├── 권한 계층 (Permission Layer)
│   ├── 레벨 1: 공용 읽기 가능  (매뉴얼, 공용 게시판)
│   ├── 레벨 2: 테넌트 내부     (고객, 업무, 메모, 일정 등)
│   └── 레벨 3: 관리자 전용     (전체 계정 정보 — 검색 대상 제외)
│
└── 검색 결과 그룹
    ├── 고객           (레벨 2)
    ├── 업무           (레벨 2)
    ├── 일정           (레벨 2)
    ├── 메모           (레벨 2)
    ├── 업무참고       (레벨 2)
    ├── 게시판         (레벨 1/2 혼합)
    └── 매뉴얼         (레벨 1)
```

### 3-2. 권한 정책

#### 고객 데이터 검색 보안 정책 (최우선)

```
규칙 1: tenant_id 완전 분리
  - 모든 고객 검색 쿼리에 반드시 WHERE tenant_id = 현재_로그인_tenant_id 조건 포함
  - 백엔드에서 JWT 토큰의 tenant_id로 강제 필터링
  - 클라이언트 파라미터로 tenant_id를 전달해도 백엔드에서 JWT 기준으로 덮어씀

규칙 2: 관리자 예외 없음
  - is_admin = true라도 통합검색 결과에서 타 사무소 고객 개인정보 노출 금지
  - 관리자의 테넌트(한우리)에 속한 고객만 검색 가능
  - 타 사무소 고객 데이터 조회는 별도 관리자 전용 기능으로만 허용 (통합검색 아님)

규칙 3: 노출 최소화
  - 고객 검색 결과에서 노출 허용 필드:
    ✅ 이름(한글), 국적, 고객ID, 등록증 만기일, 여권 만기일
  - 고객 검색 결과에서 노출 금지 필드:
    ❌ 주민번호 성격의 정보(등록증 번호 전체), 여권번호 전체, 주소, 연락처
  - 클릭 시 해당 고객의 상세 페이지로 이동 (권한 확인 후 풀 정보 표시)

규칙 4: 검색 감사 로그
  - 고객 데이터 검색 시 로그 기록: 검색자(login_id), 검색어, 검색 시각, 결과 건수
  - 로그는 별도 Audit 시트 또는 로그 파일에 append-only로 저장
  - 향후 감사/점검 시 추적 가능하도록 설계
```

#### 게시판 검색 정책

```
공용 게시판 (기본 게시판):
  - 로그인한 모든 사무소 사용자 검색 가능
  - 공지, Q&A, 사례공유, 자료실 탭 모두 포함

비공개/사무소 내부 게시글:
  - 해당 사무소(tenant)만 검색 가능
  - 추후 게시판에 "사무소 전용" 토글 추가 시 적용

검색 결과 표시 필드:
  - 제목, 본문 앞 80자, 태그, 작성일, 작성자(사무소명), 게시판 유형
  - 첨부파일 유무 표시
```

#### 기타 데이터 권한 정책

```
업무(진행/예정/완료):  tenant 격리 — 해당 사무소만 검색
일정:                 tenant 격리 — 해당 사무소만 검색
메모(단기/장기/중기): tenant 격리 — 해당 사무소만 검색
업무참고:             tenant 격리 — 각 사무소의 work_sheet_key에서만 검색
매뉴얼:               공용 — 로그인한 모든 사용자 검색 가능
문서 템플릿 메타:      공용 읽기 전용 — 민원명/서류 종류만 검색 대상
```

### 3-4. 검색 기술 구현 방식

**백엔드 검색 아키텍처:**
```
POST /api/search?q={keyword}
  → 헤더에서 JWT 추출 → tenant_id 확인
  → 병렬 검색 시작:
      ① customers 시트  (tenant_id 강제 필터)
      ② tasks/active    (tenant_id 강제 필터)
      ③ tasks/planned   (tenant_id 강제 필터)
      ④ tasks/completed (tenant_id 강제 필터)
      ⑤ events          (tenant_id 강제 필터)
      ⑥ memos           (tenant_id 강제 필터)
      ⑦ reference       (tenant 고유 work_sheet_key)
      ⑧ board           (공용 + 권한 필터)
      ⑨ manual          (공용 데이터)
  → 결과 병합 및 그룹핑
  → 감사 로그 기록
  → 응답 반환
```

**감사 로그 구조:**
```
필드: log_id | login_id | tenant_id | keyword |
      result_counts(JSON) | searched_at | ip_addr
저장: 별도 AuditLog 시트 또는 로컬 파일
```

## 7. 문서자동작성 구현 범위 확정

### 7-1. 반드시 구현해야 하는 핵심 로직

이 기능은 UI만 구현하는 것이 아니라 **Streamlit page_document.py의 실무 로직 전체**를 Next.js + FastAPI로 이식합니다.

**민원 조합 트리:**
```
CATEGORY_OPTIONS (상위 구분)
  └─ MINWON_OPTIONS (민원 종류)
      └─ TYPE_OPTIONS (신청 유형)
          └─ SUBTYPE_OPTIONS (세부 분류)
```
각 조합에 대응하는:
- `REQUIRED_DOCS`: 필요서류 목록
- `DOC_TEMPLATES`: 사용할 PDF 템플릿 파일 목록

**역할 시스템 (ROLE_WIDGETS):**
```
yin   = 신청인 (본인)
hyin  = 숙소제공자
byin  = 신원보증인
gyin  = 합산자
pyin  = 대리인 (행정사 → 자동 채움)
ayin  = 기타 관련인
```
각 역할에는 적용되는 PDF 위젯(필드명) 목록이 매핑됨.

**PDF 생성 파이프라인:**
```
1. 민원 조합 선택 → 해당 PDF 템플릿 파일들 결정
2. 역할별 인물 선택 (고객 DB 또는 행정사 정보)
3. build_field_values() → 각 역할의 데이터를 PDF 필드명에 매핑
4. make_seal_bytes() → 행정사 도장 이미지 생성 (PIL, 원형)
5. fill_and_append_pdf() → PyMuPDF로 각 PDF에 필드 주입
6. 다중 문서 생성 → PDF 병합 또는 ZIP으로 제공
7. 브라우저에서 인라인 미리보기 또는 다운로드
```

### 7-2. 백엔드 API 명세

```
GET  /api/documents/options          → 전체 민원 조합 트리 반환
GET  /api/documents/required-docs    → 선택 조합의 필요서류 목록
GET  /api/documents/templates        → 선택 조합의 템플릿 목록
POST /api/documents/generate         → PDF 생성 (멀티파트 반환 or base64)
GET  /api/admin/agent-info           → 현재 로그인 사무소 행정사 정보
```

### 7-3. 프론트엔드 5단계 스텝 (확정)

```
STEP 1: 구분 선택     → 카드형 선택 (4개 카테고리)
STEP 2: 민원 선택     → 드롭다운
STEP 3: 세부 선택     → 드롭다운 2개 (유형 + 세부)
STEP 4: 인물 선택     → 역할별 고객 검색 자동완성 + 대리인 자동채움
          ↳ 필요서류 목록 자동 표시 (체크리스트)
STEP 5: 검토 및 생성  → 입력값 확인표 + PDF 생성 버튼
          ↳ 생성 후: 인라인 PDF 미리보기 + 다운로드/인쇄
```

### 7-4. 대리인(행정사) 자동 채움 규칙

- JWT 토큰 → login_id → Accounts 시트 조회
- `office_name`, `office_adr`, `contact_name`, `biz_reg_no`, `agent_rrn` 자동 반영
- 사용자가 수정 불가 (표시 전용)

### 7-5. 도장 생성 규칙

- `create_seal()`: PIL로 원형 도장 이미지 생성
- 입력: 행정사명(contact_name)
- 출력: PNG 바이트
- PDF의 도장 필드 위치에 이미지 삽입
