==================================================
B. BACK SIDE RULE
==================================================

This is the main redesign.

Step 1: find the back-side largest inner table
- On the ARC back side, detect the largest inner rectangular table.
- This is the primary address/date table.
- Do not rely on OCR text to find it.

Step 2: use the table divider for 90-degree orientation
- Inside that largest table, detect the long vertical divider line.
- Use that divider to determine the coarse orientation among:
  - 0
  - 90
  - 180
  - 270
- The correct orientation should make the divider vertical and positioned near the left-side narrow column.

Practical divider-position rule:
- The divider is not exactly 2:3.
- Treat the correct left-column ratio roughly as 0.32 ~ 0.39 of the table width.
- The opposite reversed case will look like 0.61 ~ 0.68.
- Use ranges, not a single exact ratio.

Step 3: fine deskew using divider angle
- After coarse orientation is decided, use the divider angle itself to do fine deskew.
- Example:
  - if coarse orientation is 90 degrees
  - and the divider is still tilted by +1.8 degrees
  - rotate accordingly so the divider becomes truly vertical

Important:
- Do NOT use projection profile fallback.
- Only use:
  - largest table
  - divider position
  - divider angle
for orientation and deskew.

==================================================
D. ADDRESS EXTRACTION RULE
==================================================

Address extraction must work like this:

1. OCR the right wide address column only.
2. Do lightweight cleanup.
3. Then apply typo correction using similarity-based normalization.

Important:
- Similarity is allowed for correcting OCR typos.
- This should be hierarchical and cheap, not a full fuzzy search over all Korean addresses.

Use this order:
1. Level 1 region similarity
   - ex: 경기도, 서울특별시, 인천광역시, etc.
2. Level 2 city/district similarity
   - scoped under the selected Level 1 parent when possible
3. then keep existing road/address scoring logic as needed

Important constraints:
- Do NOT fuzzy-match the full national road-name database.
- Do NOT do a heavy nationwide fuzzy search.
- Similarity should only be used for:
  - top-level region correction
  - city/district correction
  - lightweight OCR typo recovery

Also:
- If Level 1 is weak, Level 2 may help infer the parent.
- If the final address is weak, return empty string instead of garbage.
