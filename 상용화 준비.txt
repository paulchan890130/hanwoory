# 출입국관리 시스템 상용화 최소보안 빌드 계획서

## 1. 최종결정 반영사항

이번 빌드의 확정 범위는 다음과 같다.

```text
1. 행정사 주민등록번호 암호화는 이미 구현된 것으로 보고 기존 구조 유지
2. 고객 개인정보 중 외국인등록번호 뒷자리만 암호화
3. 외국인등록번호 뒷자리는 목록에서 대량 복호화하지 않고, 상세/문서작성 등 필요한 순간에만 복호화
4. 로그인 실패 제한, 계정잠금, 감사로그, 약관/동의 절차는 상용화 최소요건으로 구현
5. 여권번호, 전화번호, 주소는 이번 1차 암호화 범위에서 제외
6. 테넌트별 데이터 분리는 반드시 유지·강화
```

이번 빌드의 목적은 대기업급 보안체계가 아니라, 외부 사무소 클로즈드 베타가 가능한 **상용화 1차 최소보안 구조**를 만드는 것이다.

# 2. 전체 빌드 목표

## 2-1. 사용자 체감 목표

사용자 입장에서는 기존과 거의 동일해야 한다.

```text
로그인 전: 데이터 접근 불가
로그인 후: 기존처럼 고객정보 조회 가능
고객상세: 외국인등록번호 전체 표시 가능
문서자동작성: 기존처럼 등록번호 자동 입력
고객목록: 불필요한 경우 등록번호 뒷자리 복호화하지 않음
```

## 2-2. 내부 저장 목표

DB 내부에서는 외국인등록번호 뒷자리를 평문으로 보관하지 않는 구조로 전환한다.

```text
기존:
등록증/front = 901010
번호/back = 1234567

변경:
등록증/front = 901010
번호_encrypted = 암호문
번호_hash = 검색용 HMAC
번호_last4 = 4567
```

# 5. Phase 1 — 암호화 유틸 설계

## 5-1. 암호화 방식

외국인등록번호 뒷자리는 서버에서 암호화한다.

권장 방식:

```text
대칭키 암호화: Fernet 또는 AES-GCM
키 보관: Render 환경변수
키 로딩: 서버 시작 시 1회 로딩
브라우저 전달: 금지
로그 출력: 금지
```

환경변수 예시:

```text
PII_ENCRYPTION_KEY
PII_HASH_SECRET
```

## 5-2. hash 방식

외국인등록번호 뒷자리는 7자리라 단순 SHA256을 쓰면 무차별 대입 위험이 있다.

따라서 검색용 hash는 단순 hash가 아니라 HMAC을 사용한다.

```text
나쁜 예:
sha256("1234567")

좋은 예:
hmac_sha256(PII_HASH_SECRET, tenant_id + ":" + "1234567")
```

## 5-3. 정규화 규칙

저장 전 등록번호 뒷자리를 정규화한다.

```text
입력값: 1-234567
정규화: 1234567

입력값: 123 4567
정규화: 1234567

입력값: 1234567
정규화: 1234567
```

유효성:

```text
빈 값이면 암호화하지 않음
숫자 7자리가 아니면 저장은 가능하되 경고 또는 validation 검토
마이그레이션 중 비정상 값은 별도 로그로 분리
```

# 6. Phase 2 — DB 마이그레이션 설계

## 6-1. 신규 컬럼 추가

실제 테이블명을 확인한 후 Alembic migration을 만든다.

예시 컬럼:

```text
alien_reg_back_encrypted TEXT
alien_reg_back_hash TEXT
alien_reg_back_last4 VARCHAR(4)
alien_reg_back_migrated_at TIMESTAMP
```

선택 컬럼:

```text
alien_reg_back_enc_version VARCHAR(20)
```

암호화 알고리즘 버전 관리가 필요하면 추가한다.

## 6-2. 기존 평문 컬럼 처리

초기 마이그레이션 단계에서는 기존 평문 컬럼을 바로 삭제하지 않는다.

1차 구조:

```text
기존 평문 컬럼 유지
신규 encrypted/hash/last4 컬럼 추가
읽기는 encrypted 우선
encrypted 없으면 legacy 평문 fallback
쓰기는 encrypted 우선
```

이유:

```text
1. 운영 DB 마이그레이션 실패 대비
2. 기존 API 호환성 유지
3. 문서자동작성 오류 방지
4. 롤백 가능성 확보
```

## 6-3. 후속 정리

충분히 검증한 뒤 2차 작업으로 기존 평문 뒷자리 컬럼을 비우거나 제거한다.

```text
1차: encrypted 컬럼 추가 + fallback
2차: 운영 검증 후 평문 null 처리
3차: 더 이상 필요 없으면 컬럼 제거 검토
```

# 7. Phase 3 — 데이터 마이그레이션

## 7-1. dry-run 스크립트

운영 적용 전 dry-run 스크립트를 만든다.

dry-run 출력 항목:

```text
전체 고객 수
등록번호 뒷자리 평문 존재 수
암호화 대상 수
이미 암호화된 수
비정상 값 수
tenant_id 누락 수
```

## 7-2. 실제 마이그레이션

실제 마이그레이션 흐름:

```text
1. 운영 DB 백업
2. Alembic migration으로 컬럼 추가
3. 데이터 암호화 스크립트 실행
4. 평문 뒷자리 → encrypted/hash/last4 생성
5. 암호화 완료 row에 migrated_at 기록
6. 결과 리포트 출력
```

## 7-3. 마이그레이션 원칙

```text
기존 평문값 즉시 삭제 금지
실패한 row는 건너뛰고 리포트
transaction 단위는 너무 크게 잡지 않기
tenant_id 없는 row는 별도 확인
로그에 원문 등록번호 출력 금지
```

# 8. Phase 4 — API 읽기/쓰기 수정

## 8-1. 고객 생성/수정 API

고객 생성 또는 수정 시:

```text
1. 입력값에서 외국인등록번호 뒷자리 추출
2. 정규화
3. encrypted 생성
4. HMAC hash 생성
5. last4 생성
6. DB 저장
7. 평문 로그 출력 금지
```

## 8-2. 고객 상세조회 API

상세조회 시:

```text
1. 로그인 사용자 tenant_id 확인
2. 고객 row의 tenant_id와 일치 여부 확인
3. 일치하면 alien_reg_back_encrypted 복호화
4. 프론트엔드에는 기존 필드명 또는 호환 필드로 반환
```

프론트엔드 호환을 위해 기존 응답 형식은 가능하면 유지한다.

예시:

```json
{
  "alien_reg_front": "<YYMMDD>",
  "alien_reg_back": "<BACK7>",
  "alien_reg_no": "<YYMMDD>-<BACK7>"
}
```

다만 내부 로그에는 원문이 남지 않게 한다.

## 8-3. 고객목록 API

목록에서는 기본적으로 복호화하지 않는다.

반환 예시:

```json
{
  "alien_reg_front": "901010",
  "alien_reg_back_masked": "1******",
  "alien_reg_back_last4": "4567"
}
```

고객목록에서 전체 등록번호가 꼭 필요하지 않다면 절대 복호화하지 않는다.

## 8-4. 검색 API

검색 방식:

```text
성명 검색: 기존 방식
전화번호 검색: 기존 방식 또는 last4
등록번호 뒷자리 전체 검색: HMAC hash 비교
등록번호 뒤 4자리 검색: last4 비교
```

주의:

```text
encrypted 컬럼에 LIKE 검색 금지
등록번호 원문을 검색 로그에 저장 금지
```

# 9. Phase 5 — 문서자동작성 연동

문서자동작성에서는 등록번호 뒷자리가 필요하다.

처리 흐름:

```text
1. 문서작성 요청
2. tenant_id 권한 확인
3. 선택 고객 row 조회
4. alien_reg_back_encrypted 복호화
5. PDF 필드값 구성
6. PDF 생성
7. 복호화 원문 로그 저장 금지
```

관련자도 동일하게 처리한다.

대상 가능성:

```text
신청인
숙소제공자
신원보증인
법정대리인
합산자
행정사
```

단, 행정사 주민번호는 기존 구현 구조를 우선 사용하고 이번 작업에서 임의 변경하지 않는다.

# 10. Phase 6 — OCR/스캔 저장 연동

OCR이나 스캔으로 고객을 추가하는 경우에도 같은 규칙을 적용한다.

처리 흐름:

```text
1. OCR 결과에서 등록번호 추출
2. 앞자리/뒷자리 분리
3. 뒷자리 정규화
4. encrypted/hash/last4 생성
5. DB 저장
```

주의:

```text
OCR 원문 텍스트 로그에 등록번호 전체가 남지 않게 한다.
디버그 화면에 원문 OCR 결과를 보여주는 경우 운영에서는 제한한다.
```

# 11. Phase 7 — 로그인 실패 제한 / 계정잠금

초기 정책:


# 14. Phase 10 — 테넌트 분리 점검

상용화에서 가장 중요한 부분이다.

점검 대상:

```text
고객 목록
고객 상세
고객 수정
고객 삭제
문서자동작성
OCR 저장
완료업무
진행업무
일정
메모
관리자 기능
파일 다운로드
```

필수 조건:

```text
모든 고객 관련 쿼리는 tenant_id 조건을 포함해야 한다.
프론트엔드에서 customer_id만 바꿔 다른 사무소 고객을 볼 수 없어야 한다.
관리자 권한과 일반 사용자 권한을 분리해야 한다.
```

테스트:

```text
A사무소 고객ID를 B사무소 계정으로 직접 요청 → 403 또는 404
B사무소 고객ID를 A사무소 계정으로 직접 요청 → 403 또는 404
관리자만 전체 조회 가능
일반 계정은 자기 tenant만 가능
```

# 18. 롤백 계획

## 18-1. 1차 롤백

초기에는 기존 평문 컬럼을 유지하므로, 문제가 생기면 API를 legacy fallback으로 되돌릴 수 있다.

```text
encrypted 읽기 실패 → legacy 평문 사용
문서자동작성 오류 → legacy 필드 사용
검색 오류 → 기존 검색 복구
```

## 18-2. DB 롤백

```text
운영 DB 백업본 복구
또는 신규 컬럼 무시
```

초기 배포에서는 기존 평문 컬럼을 삭제하지 않기 때문에 롤백 난이도를 낮출 수 있다.

# 19. 중단 조건

작업 중 아래 상황이 발견되면 즉시 중단하고 보고한다.

```text
1. 외국인등록번호 뒷자리가 여러 테이블에 중복 저장되어 있음
2. 문서자동작성에서 등록번호 필드 구조가 예상과 다름
3. OCR 저장 로직이 고객 저장 API를 우회함
4. 고객목록에서 전체 등록번호 복호화가 필수로 되어 있음
5. 행정사 주민번호 기존 암호화 키와 신규 키 구조가 충돌함
6. 운영 DB에 tenant_id 누락 고객이 존재함
7. 기존 검색이 등록번호 부분검색에 강하게 의존함
```

이 경우 임의로 구조를 변경하지 말고, 확인된 사실과 선택지를 보고해야 한다.

# 21. 최종 확정 구조

```text
상용화 1차 최소보안 구조:

- 행정사 주민등록번호: 기존 암호화 구조 유지
- 고객 외국인등록번호 뒷자리: 신규 암호화
- 고객목록: 대량 복호화 금지
- 고객상세: 선택 고객만 복호화
- 문서자동작성: 필요한 고객만 복호화
- 검색: hash/last4 사용
- 로그인: 실패 제한 및 잠금
- 감사로그: 주요 개인정보 접근만 기록
- 약관/동의: 가입 및 최초 로그인 시 기록
- 테넌트 분리: 전 API에서 재검증
```

이 구조로 진행하면 보안 부담은 최소화하면서도 외부 사무소 대상 클로즈드 베타를 시작할 수 있는 기본 방어선을 갖출 수 있다.
