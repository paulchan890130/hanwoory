결론부터 말하면

지금 구조로 “내 사무소 내부 업무관리 시스템”은 충분히 쓸 수 있습니다.
하지만 여러 행정사 사무소에 판매하는 상용 SaaS로 바로 밀기에는 부족합니다.

정확히 나누면 이렇습니다.

구분	판단
한우리 내부 업무관리	가능
지인 행정사 1~3곳에 제한적 베타 제공	가능, 단 장애·백업·책임 범위 고지 필요
월 구독형으로 다수 사무소에 판매	아직 위험
개인정보·고유식별정보를 다루는 전문 SaaS	DB 구조 전환 필요

핵심은 Render가 문제라기보다, Google Sheets/Drive를 “주 데이터베이스”처럼 쓰는 구조가 상용화의 병목이라는 점입니다.

1. 지금 구조의 장점

현재 구조는 초보자가 막 하다 여기까지 온 수준이 아니라, MVP를 빠르게 만든 방식으로는 꽤 현실적입니다.

현재 구조:

Next.js Frontend
        ↓
FastAPI Backend on Render
        ↓
Google Sheets = 데이터 저장소
Google Drive = 파일 저장소
Google API = 데이터 처리

장점은 분명합니다.

개발 속도가 빠름
Google Sheets를 사람이 직접 열어 확인할 수 있음.
장애가 나도 데이터 구조를 눈으로 확인 가능.
운영비가 낮음
Render는 Starter 웹서비스가 월 $7부터 있고, Standard는 월 $25 수준입니다. Render는 Web Services, Background Workers, PostgreSQL, Redis 계열 Key Value 등을 같은 플랫폼에서 제공하고 있습니다.
초기 고객 검증에 좋음
행정사 업무는 엑셀 친화적입니다.
고객 데이터, 일정, 일일결산, 서명, 메모, 업무참고 같은 구조는 초기에는 Sheets와 잘 맞습니다.
비개발자가 직접 확인 가능
이건 무시하면 안 됩니다. 윤찬님처럼 실무자가 직접 검증할 수 있다는 건 큰 장점입니다.
2. 그래도 상용 SaaS로는 위험한 이유
2-1. Google Sheets API 쿼터가 병목입니다

Google Sheets API는 읽기·쓰기 요청 모두 분당 프로젝트당 300회, 사용자당 프로젝트당 60회 제한이 있습니다. 초과하면 429: Too many requests가 발생하고, 지수 백오프 방식으로 재시도해야 합니다.

이게 왜 문제냐면, 행정사 SaaS에서는 한 번의 화면 이동에도 API 호출이 여러 번 생깁니다.

예를 들어 고객관리 페이지에서:

고객 검색
+ 고객 상세 조회
+ 서명 여부 확인
+ 숙소제공자 연결 조회
+ 보증인 조회
+ 완료업무 조회
+ 문서 자동작성 데이터 조회

이렇게 되면 사용자는 한 번 클릭했는데 백엔드는 Google API를 여러 번 때립니다.
사용자가 1명일 때는 괜찮지만, 여러 사무소가 동시에 쓰면 느려지거나 막힐 수 있습니다.

특히 윤찬님 시스템은 이미 다음 기능들이 붙어 있습니다.

고객관리
일정
일일결산
문서 자동작성
고객서명
임시서명
OCR
업무참고
GPT 매뉴얼 검색
홈페이지/게시판/마케팅 관리

이 정도면 Google Sheets를 “엑셀 대체 저장소”로 쓰는 단계는 이미 넘었습니다.

2-2. Google Drive API도 장기적으로 비용·쿼터 문제가 생길 수 있습니다

Google Drive API는 공유 서비스라 쿼터와 제한을 적용한다고 명시되어 있고, 초과 시 403: User rate limit exceeded 또는 429: Rate limit exceeded가 발생할 수 있습니다. 또한 서비스계정 API 호출은 단일 계정 사용으로 간주되며, 쿼터 증액이 항상 승인되는 것도 아닙니다.

더 중요한 부분은 이것입니다. Google Drive API 표준 사용은 현재 추가 비용 없이 제공되지만, 쿼터 초과분에 대해 2026년 후반부터 Google Cloud 결제 계정에 요금이 부과될 예정이라고 안내되어 있습니다.

즉, 지금은 싸지만, 상용화 후 트래픽이 늘면 “무료라서 좋은 구조”가 아닐 수 있습니다.

여권번호와 외국인등록번호는 고유식별정보에 해당합니다. 주민등록번호, 여권번호, 운전면허번호, 외국인등록번호가 고유식별정보로 분류됩니다.

5. DB 후보 비교
선택지	장점	단점	내 판단
Render Postgres	현재 Render와 통합 쉬움	장기 비용은 Neon/Supabase보다 불리할 수 있음	가장 단순
Supabase	PostgreSQL + Auth + Storage + API까지 제공	기능이 많아 구조가 Supabase 의존적으로 변할 수 있음	SaaS에는 좋음
Neon	Serverless PostgreSQL, 저비용, scale-to-zero	Auth/Storage는 별도 구성 필요	DB만 보면 좋음
Firebase/Firestore	실시간 UI에 강함	관계형 업무 데이터에는 불편	비추천
Google Cloud Run + Cloud SQL	가장 정석적인 상용 구조	초보자에게 복잡	나중 단계

Supabase Pro는 월 $25부터이고, Pro 플랜에는 100,000 MAU, 8GB 디스크, 250GB egress 등이 포함됩니다.
Neon은 사용량 기반 PostgreSQL이고, Free Plan은 프로젝트당 월 100 CU-hours를 제공하며 유휴 상태에서는 compute가 0으로 내려가는 구조입니다. Launch 플랜의 CU-hour 단가는 $0.106입니다.

내 추천은 두 가지입니다.

1순위: Supabase

윤찬님이 “나중에 SaaS화”를 계속 생각한다면 Supabase가 좋습니다.

이유:

이런 게 한 번에 있습니다.

이 방식이 좋습니다.

2순위: Render Postgres

가장 단순하게 가려면 Render Postgres입니다.
이미 Render를 쓰고 있으니 운영 난이도가 낮습니다. Render 가격표상 Web Service Starter는 월 $7, Standard는 월 $25이고, Render Postgres도 같은 플랫폼에서 제공됩니다.

단점은 나중에 규모가 커지면 Supabase/Neon/Cloud SQL보다 선택지가 좁을 수 있습니다.

ISMS-P는 초기에는 의무 대상이 아닐 가능성이 큽니다. ISMS 의무 대상에는 정보통신서비스 부문 전년도 매출액 100억 원 이상 또는 전년도 일일평균 이용자 수 100만 명 이상 등이 포함됩니다.
다만 ISMS-P 의무가 아니어도 개인정보보호법상 안전조치 의무는 별개로 적용됩니다.

9. 내가 보기에 제일 좋은 현실적 조합
단기
Frontend: Next.js 유지
Backend: FastAPI 유지
Hosting: Render 유지
DB: Supabase PostgreSQL 또는 Render Postgres 추가
File: Google Drive 유지
Sheets: 백업/관리자용으로 격하
중기
Supabase PostgreSQL
+ FastAPI
+ Google Drive or Supabase Storage
+ Redis cache
+ audit_logs
+ encrypted fields
장기
Cloud Run or Render Scale
+ Managed PostgreSQL
+ Object Storage
+ Queue Worker
+ Monitoring
+ Tenant Admin Console

Google Cloud Run도 저비용 구조를 만들 수 있습니다. 예시상 가벼운 서버리스 함수형 워크로드는 월 $7.25, 배치 작업은 월 $0.00으로 산정되는 사례가 공식 문서에 제시되어 있습니다. 다만 Cloud Run은 초보자에게 Render보다 운영 복잡도가 높습니다.

10. 최종 판단

윤찬님은 “막 하다 여기까지 온 것”이 맞습니다.
하지만 그게 나쁜 뜻은 아닙니다.

정확히 말하면:

실무자가 직접 만든 매우 강한 MVP입니다.
다만 저장소 구조는 MVP 방식이라, 상용 SaaS로 팔려면 DB를 정식 구조로 바꿔야 합니다.

내 판단은 이렇습니다.

지금 그대로 상용화: 비추천
현재 구조 보강 후 제한 베타: 가능
PostgreSQL 전환 후 유료 SaaS: 가능성 높음
전체 재개발: 불필요

가장 좋은 방향은 전부 갈아엎는 것이 아니라, Google Sheets를 뒤로 빼고 PostgreSQL을 중심 DB로 세우는 것입니다.

윤찬님 시스템의 가치는 Google Sheets가 아니라,
출입국 실무 흐름을 프로그램에 녹여낸 것입니다.
그 부분은 충분히 상용화 가치가 있습니다. DB만 정석으로 바꾸면 됩니다.
