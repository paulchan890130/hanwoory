# 실무자 연락처 · 문서 대표정보 설계 노트

관련 브랜치: `fix/selfcheck-visibility-and-document-office-profile`

## 현재 사실 (2026-07-24 기준)

- **가입신청(`/apply`)에는 실무자(office_staff) 전화 입력란이 없다.** 신청은 대표자(주계정)
  중심으로 접수되며, 실무자 계정은 승인 후 발급된다.
- **실무자 본인 연락처(`users.contact_tel`)는 실무자 개인 값**이다. 실무자가 마이페이지에서
  `PATCH /api/auth/me`(`contact_tel`)로 직접 저장하며, 저장 즉시 서버 재조회로 확정한다.
- **문서 자동작성에는 실무자 연락처를 사용하지 않는다.** 문서에는 tenant 공통정보(사무소명·
  주소·사업자번호·행정사 주민등록번호) + **대표자 계정의 `contact_name`/`contact_tel`** 만
  들어간다. 대표자 = 활성(is_active) 관리자(is_admin) 중 id 최소(결정적). 대표자가 없으면
  실무자로 fallback하지 않고 `find_document_office_profile_by_tenant_pg` 가
  `representative_configured=False` 로 표시 → quick_doc 에서 **409
  `OFFICE_REPRESENTATIVE_NOT_CONFIGURED`** 로 차단한다.

## "실무자 전화 자동 반영"이 제품 요구가 될 경우 (설계 제안 — 미구현)

문서에 실무자 전화를 넣는 것은 **현재 계약과 상충**한다(문서는 대표 행정사 명의). 만약 특정
문서에서 "작성 담당 실무자 연락처"를 별도 필드로 노출해야 한다면:

1. 문서 필드 계약에 대표자 필드(`agent_tel`)와 **구분되는** 별도 필드(예: `staff_tel`)를
   신설하고, 그 필드를 쓰는 템플릿에서만 현재 로그인 실무자 연락처를 채운다. 대표자 계약은
   불변.
2. 가입신청에 실무자 전화 입력을 추가하려면 신청 폼 + 승인/활성화 흐름 + 저장 스키마를
   함께 설계해야 한다. **이번 브랜치에서는 임의 migration 을 추가하지 않는다** — 실제 필드
   계약과 제품 요구가 확정된 뒤 별도 브랜치에서 진행한다.

현재로서는 실무자 전화의 용도는 **실무자 본인 식별/연락**에 한정되며 문서 출력과 무관하다.
